# SBI Cards – Kong Enterprise CI/CD Demo

A complete CI/CD pipeline demonstrating how to design, deploy, and test a mocked API on **Kong Enterprise** using an OpenAPI spec, decK, and Insomnia.

The scenario: SBI Cards exposes a **Reward Points API**. The API is mocked on Kong using the Kong Mocking plugin so it returns realistic responses without a backend service — ideal for parallel frontend/consumer development.

---

## What this repo demonstrates

| Step | Tool | What happens |
|------|------|--------------|
| 1 | `inso lint` | Validate the OpenAPI spec |
| 2 | `deck file openapi2kong` | Convert OAS → Kong declarative config |
| 3 | `deck file add-plugins` | Inject the Mocking + CORS plugins into every service |
| 3 | `deck gateway apply` | Push config to Kong Enterprise |
| 4 | Admin API `/files` + `/document_objects` | Publish the spec to the Kong Dev Portal |
| 5 | `inso run test` | Run automated API tests against the mock |

## Running locally

```bash
# Clone the repo
git clone https://github.com/siddsethi11/sbicards-kong-cicd.git
cd sbicards-kong-cicd

# Set environment (adjust to your Kong instance)
export KONG_ADMIN_URL="http://localhost:8001"
export KONG_PROXY_URL="http://<kong_proxy_url>"
# export KONG_ADMIN_TOKEN="<your-rbac-token>"   # only if RBAC is on

# Full 5-step run
./test-local.sh

# Or run individual steps
./test-local.sh --lint-only     # Step 1: lint the OAS spec
./test-local.sh --deploy-only   # Steps 2–4: convert, deploy, publish to Dev Portal
./test-local.sh --test-only     # Step 5: run API tests
./test-local.sh --skip-deploy   # Steps 1–3 local build/validate only (no Admin API calls)
```

## Testing the mock API manually

Once deployed, hit the mock endpoint directly:

```bash
# Returns mocked reward points response (no backend required)
curl "${KONG_PROXY_URL}/cards/4242/rewards"
```

Expected response:
```json
{
  "cardNumber": "4242",
  "cardHolder": "Rajesh Kumar",
  "rewardPoints": 12500,
  "tier": "Gold",
  "pointsExpiryDate": "2024-12-31",
  "equivalentCashValue": 125.00
}
```

Quick CORS preflight check:

```bash
curl -i -X OPTIONS "${KONG_PROXY_URL}/cards/4242/rewards" \
  -H "Origin: http://localhost:8003" \
  -H "Access-Control-Request-Method: GET" \
  -H "Access-Control-Request-Headers: Content-Type,Authorization,X-API-KEY"
```

Expected headers include:

```text
access-control-allow-origin: http://localhost:8003
access-control-allow-methods: GET,HEAD,POST,PUT,PATCH,DELETE,OPTIONS
```

## GitHub Actions CI/CD

The workflow (`.github/workflows/kong-ee-cicd.yml`) runs on a **self-hosted runner** with `deck` pre-installed and network access to the Kong Admin API.

| Job | Trigger | Steps |
|-----|---------|-------|
| `lint-openapi` | PR + push to `main` | Installs inso, lints OAS |
| `deploy-kong-ee` | Push to `main` only | Converts, deploys, publishes to Dev Portal |
| `test-with-inso` | Push to `main` only | Runs Insomnia test suite against the proxy |

### Required GitHub configuration

**Repository secrets** (`Settings → Secrets and variables → Actions → Secrets`):

| Secret | Value |
|--------|-------|
| `KONG_ADMIN_TOKEN` | Kong RBAC token (if RBAC is enabled) |
| `KONG_PROXY_URL` | Kong proxy URL for test runner |

**Repository variables** (`Settings → Secrets and variables → Actions → Variables`):

| Variable | Example |
|----------|---------|
| `KONG_ADMIN_URL` | `http://kong-admin.internal:8001` |
| `KONG_WORKSPACE` | `default` |

## How plugins work

The Kong [Mocking plugin](https://docs.konghq.com/hub/kong-inc/mocking/) intercepts requests and returns example responses directly from the embedded OpenAPI spec — no upstream service needed.

The Kong [CORS plugin](https://docs.konghq.com/hub/kong-inc/cors/) is also applied at the service level to allow browser-based calls from local UIs (for example `http://localhost:8003`).

`kong/plugins.yaml` is a decK add-plugins manifest that injects both plugins on every service generated from the OAS spec:

```yaml
add-plugins:
  - selectors:
      - "$..services[*]"
    plugins:
      - name: cors
        config:
          origins:
            - http://localhost:8003
      - name: mocking
        config:
          api_specification: |
            # (full OAS spec embedded here)
```

If you want to apply a standalone CORS plugin payload directly against Kong Admin API, you can use:

```bash
cat <<'EOF' | deck gateway apply - --kong-addr http://localhost:8001
_format_version: "3.0"
_workspace: default
plugins:
  - name: cors
    id: af8f03bf-9ed2-4432-b235-56d1f1f6c6b4
    enabled: true
    protocols:
      - grpc
      - grpcs
      - http
      - https
    config:
      private_network: false
      max_age: 3600
      origins:
        - http://localhost:8003
        - http://127.0.0.1:8003
      preflight_continue: false
      methods:
        - GET
        - HEAD
        - POST
        - PUT
        - PATCH
        - DELETE
        - OPTIONS
      exposed_headers:
        - X-Request-Id
        - X-API-KEY
        - apikey
      credentials: false
      allow_origin_absent: true
      headers:
        - Accept
        - Content-Type
        - Authorization
        - X-API-KEY
        - apikey
EOF
```

The `x-kong-route-defaults` extension in the OAS spec ensures generated routes accept both HTTP and HTTPS:

```yaml
x-kong-route-defaults:
  protocols:
    - http
    - https
```
